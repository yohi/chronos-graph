# Context Store MCP v2.0 アーキテクチャレビューおよび改善提案

## 1. アーキテクチャ概要と総合評価

本設計は、AIエージェントの長期記憶を管理するMCPサーバーとして、非常に堅牢かつ実用的なアーキテクチャを備えています。処理をIngestion（取り込み）、Retrieval（検索）、Lifecycle（ライフサイクル）の3つの独立したパイプラインに分離するアプローチは、保守性とテスト容易性を高める優れた選択です。また、フルモード（PostgreSQL+Neo4j+Redis）とライトウェイトモード（SQLite）を両立させ、障害時のGraceful Degradation（縮退運転）やイベント駆動型のクリーンアップ処理を組み込んでいる点は、MCPの運用特性（stdioモードでの一時的プロセス）を深く理解したエンタープライズ水準の設計として高く評価できます。

## 2. 優れている点 (What looks good)

- **パイプライン指向の明確な責務分離**: 処理フェーズが完全に分離されており、コンポーネントごとのProtocolベースの抽象化により、将来的な拡張やデータストアの切り替えが容易になっています。
- **実用的な運用モデルへの適応**: APSchedulerのような常駐スケジューラを排除し、保存回数や経過時間に基づくイベント駆動・遅延実行型のLifecycle管理を採用している点は、プロセスの寿命が短いMCPサーバーの実態に極めて即しています。
- **Graceful Degradationの組み込み**: Neo4jやRedisなどの依存サービスがダウンした際にも、システム全体を停止させずコア機能（ベクトル・キーワード検索）を継続して提供できるフォールバック設計が秀逸です。
- **トークン消費ゼロの原則**: Ingestionパイプラインにおける記憶種別の分類やクエリ意図の解析にLLMを用いず、ルールベースおよびベクトルベースのアプローチを採用することで、エージェントの運用コストの肥大化をシステムレベルで防いでいます。

## 3. リスクと改善点 (Risks & Issues)

- **Storage Layer**: PostgreSQL（マスター）とNeo4j（グラフ）への分散書き込み時のデータ整合性。片方の保存が成功し、もう片方が失敗した場合のロールバックやリトライ、結果整合性の担保メカニズムが明記されておらず、データ不整合が蓄積するリスクがあります。
- **Ingestion Pipeline (Graph Linker)**: 新規記憶の取り込み時に `SEMANTICALLY_RELATED` エッジを推定する際、類似度検索が発生しますが、データ量が増大した際に検索とエッジ生成のオーバーヘッドが蓄積し、記憶保存レイテンシの目標値（< 1,000 ms）を達成できなくなるリスクがあります。
- **MCP インターフェース (memory_save_url)**: セキュリティ（SSRF対策）は考慮されていますが、最大30秒のタイムアウトを伴う外部HTTPリクエストが同時に多数発生した場合、FastMCPの非同期ワーカーリソースが枯渇し、他の軽量な検索ツールなどの応答をブロックしてしまう懸念があります。
- **SQLite Storage**: WALモードやタイムアウト設定等のチューニングは適切ですが、複数のエージェントから同時に大量の書き込み（特に再帰的CTEを伴うグラフ更新やLifecycleのクリーンアップ）が走った場合、SQLite特有のデータベースロック（`SQLITE_BUSY`）によるエラーが多発する可能性があります。

## 4. 不足情報・確認事項 (Open Questions)

- 複数データストア（PostgreSQLとNeo4j）をまたぐ書き込みにおいて、部分的な保存失敗が発生した場合、トランザクションのロールバックや不整合データの修復はどのように行われる想定ですか？
- `Graph Linker` における `SEMANTICALLY_RELATED` エッジ作成時の類似ノード検索は、パフォーマンス維持のために検索範囲や作成されるエッジの最大数に上限を設ける仕様はありますか？
- `memory_save_url` などの外部ネットワーク通信を伴う処理が同時に多数リクエストされた場合、システム全体のレスポンス低下を防ぐための並行実行数制限（セマフォ等）やキューイングの仕様は定義されていますか？

## 5. 次のアクション (Next Steps)

- **書き込み整合性プロセスの策定**: Storage LayerにおけるPostgreSQLとNeo4jの書き込みを一つのUnit of Workとして扱う仕組み、またはLifecycle Managerに「孤立したノードや不整合なエッジの検知・修復ジョブ」を追加する。
- **並行処理・レートリミットの導入**: `memory_save_url` 等の重い外部通信処理に対して、`asyncio.Semaphore` を用いた同時実行数の制限（Concurrency Limit）を仕様と実装計画に追加する。
- **Graph Linkerのクエリ最適化**: 新規ノード追加時のエッジ自動生成において、検索対象を絞り込む条件（例: HNSWインデックスを活用したTop-K件に絞った上での類似度評価）を明確化し、パフォーマンス目標を担保する。
- **同時アクセスストレステストの追加**: 実装計画（Phase 9）のエンドツーエンド統合テストにおいて、複数プロセスからの並行書き込みをシミュレートするストレステストのタスクを追加し、SQLiteのロックタイムアウトの妥当性を検証する。
